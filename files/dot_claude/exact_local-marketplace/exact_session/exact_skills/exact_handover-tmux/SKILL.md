# Handover in a new tmux pane

現在の会話をハンドオーバー文書にまとめ、ファイルに書き出し、tmux の新しいペインでまっさらな Claude Code セッションを起動して引き継ぐ。

## 手順

### 1. 会話内容の整理

これまでの会話を以下の5観点で整理する:

- **最終ゴール**: ユーザーが最終的に達成したいこと。手段ではなく目的を書く。
- **セッション中の指示・制約**: ユーザーがセッション中に明示的に伝えた作業上のルール、好み、制約。例：「コードブロックは使わず自然言語で表現して」「〜のスタイルで書いて」「〜は避けて」など。CLAUDE.md や rules に書かれていない、このセッション固有の指示を漏らさず記録する。これがないと、前のセッションで説明した内容を新セッションのエージェントが知らずに同じ指摘を受けることになる。
- **経緯**: ここに至るまでに何を試し、何を決め、何が分かったか。完了した作業、変更したファイル、調査で判明した事実を含む。
- **現在地**: 今この瞬間、何が完了していて何が未完了か。未解決の問題は何か。
- **次の選択肢**: 次にやり得ることの候補を列挙する。「次にやること」を断定しない。判断はユーザーと新セッションのエージェントに委ねる。

#### チェーンハンドオーバーの検出

現セッションが過去のハンドオーバー文書から始まっていた場合（会話の最初のユーザーメッセージが `.844196/handovers/<datetime>_<kebab>.md` を `@` で参照している。パスは絶対・相対のどちらの形もありうる）、その前のハンドオーバー文書を Read で読む。**「最終ゴール」と「セッション中の指示・制約」は原則として前のハンドオーバー文書からそのまま引き継ぐ**。新しい情報でゴールや制約が変わった場合のみ更新する。これによりハンドオーバーを N 回チェーンしてもゴールや指示が薄れない。

ただし「参考」セクションの `前のセッション` は **絶対に引き継がない**。常に「現セッション (このハンドオーバー文書を書き出しているセッション)」の ID を入れる。新セッションから見たときの「前のセッション」 = この文書を作ったセッションを指すべきだから。前のハンドオーバー文書の値をそのままコピーすると、N 回チェーンしても全部同じ ID になり、チェーンの追跡ができなくなる。

### 2. ハンドオーバー文書の作成

書き出し先のパステンプレート (バンドルされた `handover-path.sh` がリポジトリルート解決と `mkdir -p` まで済ませたうえで返す):

!`handover-path.sh`

文書の構造:

```markdown
# <タイトル>

## 最終ゴール
...

## セッション中の指示・制約
...
<!-- このセクションがない、または該当なしの場合は省略可 -->

## 経緯
...

## 現在地
...

## 次の選択肢
...

## 参考
- 前のセッション: `${CLAUDE_SESSION_ID}`  <!-- スキルのレンダリング時に現セッション ID へ自動置換される。チェーンハンドオーバーでも前のハンドオーバー文書の値はコピーしないこと -->
- 前のハンドオーバー文書: `<前のハンドオーバー文書のパス>`  <!-- チェーンハンドオーバーの場合のみ -->
```

`.844196/` は gitignore されているのでリポジトリには残らない。ユーザーが中身を確認したい場合もパスが固定なので見つけやすい。

### 3. tmux の新しいペインで新セッションを起動

以下のコマンドを実行する（`<handover-file>` は手順2で生成したパス）:

```bash
claude-yolo --split-pane '@<handover-file> を読んで理解をユーザーに共有し、確認を待ってから動く。不明な詳細は session:session-analyzer エージェントを spawn して前セッションのトランスクリプトを確認する。'
```

新セッションは引き継ぎ内容をいきなり実行せず、まずユーザーに理解を共有して確認を待つ。これにより、捏造された次ステップが問答無用で走ってユーザーが慌てて Esc を押す事態を防ぐ。

コマンド実行後、「tmux の新ペインに引き継いだ」とだけ報告する。

## 注意事項

- ハンドオーバー文書は別セッションからの参照を前提とする。文脈を共有しない読者が理解できるよう、省略せずに記述する。
- 会話中にコードの変更を行っていた場合、その変更内容も「経緯」に含める。
- 「最終ゴール」は手段や中間成果物ではなく、ユーザーが本当に達成したいことを書く。チェーンハンドオーバーを重ねてもゴールが薄れないよう、前のハンドオーバー文書から引き継ぐことを優先する。
- 「セッション中の指示・制約」は会話の連続性を保つために重要。ユーザーが「〜しないで」「〜のスタイルで」と伝えた内容を漏らすと、新セッションで同じ指摘を繰り返させることになる。
