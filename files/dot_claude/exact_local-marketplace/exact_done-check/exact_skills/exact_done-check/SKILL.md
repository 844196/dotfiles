# Done Check

ユーザータスクを完了したと確信したターンの返答直前に、プロジェクト定義のチェックリストを走らせる Skill。

## やること

`Agent` ツールを `subagent_type: done-check:done-check` で 1 度だけ呼ぶ。`prompt` 引数には下記 4 セクションを含む markdown を組み立てて渡す。サブエージェントはメイン会話の履歴を見ないので、評価判断に必要な事実はすべてこの prompt 内に含める。

### prompt に含める 4 セクション

```
## ユーザーのゴール
<ユーザーが今回のタスクで達成したいこと。
- ハンドオーバー文書がある場合: そこに書かれているゴール記述を該当箇所だけ引用 or 要約。
- 単発の指示の場合: ユーザーの指示文をそのまま転記。
- ゴールが明示されていない場合: 「明示なし」と書いた上で、メイン会話から読み取ったタスクの趣旨を併記。>

## 実施した計画と作業
<ゴール達成のためにメインエージェントが立てた方針と、実際に行った変更・操作の要点。
編集したファイルパス、走らせたコマンド、呼び出した外部ツールも箇条書きで含める。
ファイル編集を伴わない作業 (設定変更、調査、外部 API 呼び出しのみ等) の場合もその旨を書く。>

## 達成状況の自己評価
<完了したと判断する根拠を 1〜数行で。
残課題・未着手・後回しにした懸念があれば併せて明記する。>

## スコープ判断
<ユーザーの明示指示と実際の作業範囲の照合。
- 完全一致 (除外も拡大も解釈変更もなし) なら「完全一致」と書く。
- 不一致がある場合は項目ごとに以下を明記:
  - どの項目を縮小・拡大・解釈変更したか (依頼範囲とのズレを具体的な対象名で)
  - その判断についてユーザー確認を取ったか (取った場合はそのターンの要旨、取らずに進めた場合はその旨を正直に書く)>
```

各セクションは省略不可。サブエージェントが評価対象を絞り込むうえで必要な事実 (ファイルパス、変更点、根拠) は冗長を避けつつ落とさない。逆に、サブエージェント自身がチェックリストから読み取れる内容 (チェック項目の文言など) を再掲する必要はない。

## 結果の扱い

返ってきた text を要約・加工せず、そのままチェック結果として扱う (サブエージェント側で既に圧縮された結果が返ってくる前提)。返答への反映は内容で分岐する:

- `skipped` または `全チェック対象外` 系の返答 → 返答テキストには含めず、通常の返答に進む。`Agent` ツール呼び出し自体は UI に出ているので「走った上で問題なし」はユーザーから見える
- 軽微な指摘 (typo / 文体 / 明らかな抜け漏れ等) → 自分で修正してから返答する。返答には「done-check の指摘を反映済み」と一言添える
- 方針判断が必要な指摘 → 修正は実行せず、指摘内容を返答に含めてユーザーに諾否を仰ぐ
