# Done Check

ユーザーから受けたタスクを「完了した」とエージェントが確信したタイミングで、プロジェクトが定義したチェックリストをモデルに自発的に通させる Skill + Sub-Agent + SessionStart 告知構成のプラグイン。チェック内容は `<repo-root>/.claude/done-check.{local.,}md` で自由に定義でき (テスト実行、型再生成、リント、ドキュメント整合性確認など)、設定が無ければビルトイン (ドキュメント整合性チェック) が走る。

## What It Does

タスクを完了したとエージェントが自己判断したターンで、返答テキストを書き始める直前に Skill が自発的に呼ばれる。中間報告・追加情報の要求・次のステップ宣言のような作業途中のターン、およびファイル変更を一切伴わない純粋な質問への回答のみのターンはトークン浪費を避けるためスキップする (description のトリガー語と SessionStart の事前告知を二重で効かせる、詳細は後述「Skill 起動の促し方」)。全体の流れは:

1. Skill (`done-check`) が起動。本体はメイン会話から **「ユーザーのゴール」「実施した計画と作業」「達成状況の自己評価」** の 3 セクションを組み立て、`Agent` ツール経由でサブエージェントに `prompt` として渡すディスパッチャ (詳細は後述「サブエージェントへ渡す判断材料」)
2. サブエージェント (model: sonnet) が独立した context で `bin/read-checklist.sh` を Bash 実行し、個人 → 共有 → ビルトインの順に最初に見つかったチェックリストを取得
3. 渡された 3 セクションを使ってまずゴール達成度を評価し、続けて frontmatter の `enabled` / `checks[]` と照合し該当する `checks[*].body` を評価する
4. 結果を簡潔な text にまとめて返す。先頭に `goal-fit: <結論>` を 1 行常時出力し、その下に該当した `<id>: <結論>` を列挙する形式。メイン会話の context に積まれるのはこの返答 text のみ

## How It Works

### Skill 起動の促し方

Skill ディスパッチャが取りこぼしてもチェックが走るよう、二段構えで起動を促している:

- **Skill description のトリガー**: description に「修正しました」「実装しました」「対応しました」「完了しました」「以上です」「できました」等の完了宣言フレーズを並べてあり、エージェントが完了宣言を書こうとした瞬間にマッチして自発的に Skill を呼ぶ想定。明示的な完了宣言ではなく「依頼された作業を一通り片付けた」と判断したケースもトリガー対象として併記している。
- **SessionStart の事前告知**: `hooks/session-start.sh` が `additionalContext` として「タスク完了の確信時に必ず `done-check` スキルを呼ぶ」旨をセッション冒頭に注入する。プラグインは `paths` 付き rule を提供できないため、`new-file-protocol` と同じく SessionStart の `additionalContext` を「ルール相当の事前告知」として代替する。`startup` / `resume` / `clear` / `compact` のすべての source で発火させ、resume 時にも告知が再注入される。

description マッチが先に効けばそのまま Skill が起動し、漏れた場合のフェイルセーフとして SessionStart 告知が効く。逆に作業途中のターン (中間報告、追加情報の要求、次のステップ宣言) と、ファイル変更を一切伴わない質問回答のみのターンは、トークン浪費抑制のため明示的にスキップ対象として description / additionalContext の双方に書いてある。

### 評価本体のサブエージェント分離

Skill 本体に評価ロジックを置くと、チェックリスト全文・サブエージェントが行うファイル参照・評価過程の reasoning がすべてメイン会話の context に積まれる。これを避けるため、Skill は `Agent` ツールで `subagent_type: done-check:done-check` を呼ぶだけのディスパッチャに留め、実評価は `agents/done-check.md` のサブエージェントに委譲する。

`subagent_type` がスキル名と同じ `done-check:done-check` という見た目になっているのは、プラグイン同梱エージェントの解決には `<plugin-name>:<agent-name>` 形式が必要なため (`done-check` 単独では解決されない)。たまたま plugin 名と agent 名が揃っているだけで、別名にすれば `done-check:foo` のような形になる。

- **モデル**: サブエージェントの frontmatter で `model: sonnet` を固定。メイン会話のモデル (Opus 等) より軽量・高速・安価
- **context 分離**: サブエージェントは独立した context window を持ち、その内部の Bash 出力・思考・ツール呼び出しはメイン会話に流入しない。返ってきた最終 text のみがメイン会話の context に積まれる
- **tools の制限**: サブエージェントには `Bash / Read / Grep / Glob` のみ持たせ、`Edit / Write / NotebookEdit` は持たせない。チェックは「指摘まで、修正はメイン会話側に委ねる」設計
- **preprocessing 不可**: カスタムエージェントの body では Skill のような ` ```! ` プリプロセッシングが**動かない** (記法はそのまま literal text として渡る)。そのためサブエージェント側は起動時に Bash を明示的に呼んでチェックリストを取得する

### サブエージェントへ渡す判断材料

サブエージェントは独立 context で起動するためメイン会話の履歴を一切見られない。チェック評価に必要な情報はディスパッチャ (Skill 本体) が以下 3 セクションを含む markdown を組み立てて `Agent` ツールの `prompt` に詰めて渡す。サブエージェントはこの prompt を一次情報として `checks[*].when` と照合する。

| セクション | 内容 |
|---|---|
| `## ユーザーのゴール` | 今回のタスクでユーザーが達成したいこと。ハンドオーバー文書があるならゴール記述を引用・要約、単発指示ならユーザーの指示文をそのまま転記、明示されていなければ「明示なし」と書いた上でメイン会話から読み取った趣旨を併記 |
| `## 実施した計画と作業` | ゴール達成のためにメインエージェントが立てた方針と、実際に行った変更・操作の要点。編集したファイルパス・走らせたコマンド・呼び出した外部ツールも箇条書きで含める。ファイル編集を伴わない作業 (設定変更、調査、外部 API 呼び出しのみ等) もその旨を明記する |
| `## 達成状況の自己評価` | 完了したと判断する根拠。残課題・未着手・後回しにした懸念があれば併せて明記する |

`git status` のような「リポジトリのファイル差分」を一次情報にしない理由:

- `git` 管理外の操作 (外部 API 呼び出し、データ投入、設定変更のみ等) もチェック対象に含めたい
- メイン会話の文脈そのもののほうが、ゴール達成の意味・残課題まで含めて表現できる
- サブエージェント側で `git status` を取ると、その出力がサブエージェント context のトークンを食う

副次的に「削除されたパスの存否確認」のようなニーズが出た場合は、サブエージェントが `Read` / `Glob` で対象パスを直接調べる (Bash の git コマンドには依存しない)。

### ルールの読み込み

サブエージェントが起動時に `${CLAUDE_PLUGIN_ROOT}/bin/read-checklist.sh` を Bash で実行する。スクリプトは下記の優先順位で最初に見つかったファイルを返すが、出力フォーマットは常に YAML frontmatter + markdown body の "ルール" で統一されている。そのためサブエージェントは分岐ロジックを持たず、`enabled` / `checks[]` を読むだけでどのケースも扱える。

| 優先度 | パス | 用途 |
|---|---|---|
| 1 | `<repo-root>/.claude/done-check.local.md` | 個人専用 (gitignore 推奨)。共有設定を一時上書きしたいときに使う |
| 2 | `<repo-root>/.claude/done-check.md` | プロジェクト共有設定。リポジトリにコミットしてチームで共有する |
| 3 | `references/default-checklist.md` | ビルトイン (ドキュメント整合性チェック) |

### 設定上書き

`<repo-root>/.claude/done-check.md` (共有) または `<repo-root>/.claude/done-check.local.md` (個人) に YAML frontmatter + markdown body を置くとプロジェクトごとに挙動を変えられる:

```markdown
---
enabled: true
checks:
  - id: e2e
    when: "ゴール達成のための作業内容に src/ または tests/ 配下の変更が含まれていた"
    body: |
      `mise run test:e2e` を実行し、結果を要約してから完了宣言する。
---
```

`enabled: false` で全停止。設定ファイルを置いた時点でそれより低い優先度のものは完全に上書きされる (ビルトインや共有設定の上にカスタム項目を足したい場合は、上位ファイルをひな形にして書き換える)。

## Configuration

このプラグインを有効化するには、Claude Code の `settings.json` で `enabledPlugins` に追加する:

```json
{
  "enabledPlugins": {
    "done-check@local": true
  }
}
```
