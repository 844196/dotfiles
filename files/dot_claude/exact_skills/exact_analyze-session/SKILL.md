# セッションログ分析

指定されたセッションIDのツール利用ログを分析し、改善に役立つ知見を抽出する。

## 使い方

```
/analyze-session <session-id> [session-id...] [自然言語の指示]
```

引数はセッションID（UUID形式）と自然言語の指示が混在してよい。UUID形式（`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）のトークンをセッションIDとして扱い、それ以外を分析の重点指示として解釈する。

例:
- `/analyze-session c3007db3-1a56-4f0b-b527-997f3ce8b4cb` — デフォルトの観点で分析
- `/analyze-session ツール利用がコケてるぽい c3007db3-1a56-4f0b-b527-997f3ce8b4cb` — エラー周りを重点的に
- `/analyze-session c3007db3-... 2705ba3b-... ユーザー介入が多そう` — 複数セッション、介入パターンに注目

自然言語の指示がある場合、「2. 分析の観点」のデフォルト項目に加えて、指示された観点を優先的に深掘りする。

## 手順

### 1. セッションログの取得

```bash
# パス解決（セッションIDからJSONLファイルのパスを出力する）
"$SKILL_DIR/scripts/resolve-session-path" <session-id>

# サマリー（ツール利用統計、エラー率、ユーザーメッセージ一覧）
"$SKILL_DIR/scripts/session-summary" <jsonl-path>

# ツールコール時系列（全件）
"$SKILL_DIR/scripts/extract-tool-calls" <jsonl-path>

# エラーのみ（失敗内容付き）
"$SKILL_DIR/scripts/extract-tool-calls" <jsonl-path> --errors-only --with-output
```

複数セッションIDが渡された場合は、それぞれについて上記を実行する。

### 2. 分析の観点

以下の観点でログを読み、パターンを抽出する:

- **繰り返し失敗**: 同じツール・同じパターンで複数回失敗していないか
- **ユーザー介入**: ユーザーがリジェクトした操作とその理由（is_errorのcontentにユーザーのコメントが含まれる）
- **リカバリーパターン**: 失敗後にどう立て直したか（成功したか、別の失敗に陥ったか）
- **非効率な探索**: 同じ情報を何度も読んでいないか、もっと直接的な方法がなかったか

### 3. 結果の報告

分析結果を以下の構造で報告する:

1. **セッション概要**: 何をやろうとしていたセッションか（ユーザーの最初のメッセージから判断）
2. **統計**: ツールコール数、エラー率、ユーザー介入数
3. **主要な問題パターン**: 具体的なツールコールを引用しながら
4. **改善提案**: スキル・ドキュメント・ツールの改善案
