# Handover in a new tmux pane

現在の会話をハンドオーバー文書にまとめ、ファイルに書き出し、tmux の新しいペインでまっさらな Claude Code セッションを起動して引き継ぐ。

## 手順

### 1. 会話内容の整理

これまでの会話を以下の4観点で整理する:

- **最終ゴール**: ユーザーが最終的に達成したいこと。手段ではなく目的を書く。
- **経緯**: ここに至るまでに何を試し、何を決め、何が分かったか。完了した作業、変更したファイル、調査で判明した事実、ユーザーが示した判断や好みを含む。
- **現在地**: 今この瞬間、何が完了していて何が未完了か。未解決の問題は何か。
- **次の選択肢**: 次にやり得ることの候補を列挙する。「次にやること」を断定しない。判断はユーザーと新セッションのエージェントに委ねる。

#### チェーンハンドオーバーの検出

現セッションが過去のハンドオーバー文書から始まっていた場合（会話の最初のユーザーメッセージが `@.agents/handovers/...` で始まっている）、その前のハンドオーバー文書を Read で読む。**「最終ゴール」セクションは原則として前のハンドオーバー文書からそのまま引き継ぐ**。新しい情報でゴールが変わった場合のみ更新する。これによりハンドオーバーを N 回チェーンしてもゴールが薄れない。

### 2. ハンドオーバー文書の作成

`.agents/handovers/<datetime>_<kebab-case>.md` に Write ツールで新規作成する。

`<datetime>` は `date +%Y%m%d-%H%M%S` の出力（例: `20260311-123456`）、`<kebab-case>` はテーマを端的に表すケバブケースとする（例: `20260311-123456_migrate-to-esm.md`）。

文書の構造:

```markdown
# <タイトル>

## 最終ゴール
...

## 経緯
...

## 現在地
...

## 次の選択肢
...

## 参考
- 前のハンドオーバー文書: `<前のハンドオーバー文書のパス>`  <!-- チェーンハンドオーバーの場合のみ -->
```

`.agents/` は gitignore されているのでリポジトリには残らない。ユーザーが中身を確認したい場合もパスが固定なので見つけやすい。

### 3. tmux の新しいペインで新セッションを起動

以下のコマンドを実行する（`<handover-file>` は手順2で生成したパス）:

```bash
claude-yolo --split-pane '@<handover-file> を読んで、内容の理解をユーザーに要約して伝え、確認を待ってから動き出して'
```

新セッションは引き継ぎ内容をいきなり実行せず、まずユーザーに理解を共有して確認を待つ。これにより、捏造された次ステップが問答無用で走ってユーザーが慌てて Esc を押す事態を防ぐ。

コマンド実行後、「tmux の新ペインに引き継いだ」とだけ報告する。

## 注意事項

- ハンドオーバー文書は別セッションからの参照を前提とする。文脈を共有しない読者が理解できるよう、省略せずに記述する。
- 会話中にコードの変更を行っていた場合、その変更内容も「経緯」に含める。
- 「最終ゴール」は手段や中間成果物ではなく、ユーザーが本当に達成したいことを書く。チェーンハンドオーバーを重ねてもゴールが薄れないよう、前のハンドオーバー文書から引き継ぐことを優先する。
