# Handover

現在の会話で扱ったテーマ・調査結果・進捗・未解決事項をハンドオーバー文書にまとめる。

## ワークフロー

### 1. 会話内容の整理

これまでの会話を振り返り、以下の観点で整理する:

- 何を解決・実現しようとしていたか
- どこまで進んだか（完了した作業、変更したファイル）
- 調査で判明した事実
- 未解決の問題や残タスク
- ユーザーが示した判断や好み

### 2. ハンドオーバー文書の作成

`.agents/handovers/<kebab-case>.md` を作成する。

ファイル名はテーマを端的に表すケバブケースとする（例: `migrate-to-esm.md`, `improve-build-performance.md`）。

文書のフォーマット:

```markdown
# <タイトル>

## 背景

- 何を解決・実現したいのか
- 現状の問題点

## 調査結果

調査で判明した事実を記述する。コードの該当箇所、外部ドキュメントの要点など。

## 関連するコード箇所

引き継ぎ先が探索を始める起点となるファイルパスをリストアップする。

- `path/to/file.ts` — 役割や関連する理由の簡潔な説明
- `path/to/config.json` — 同上

## 完了した作業

このセッションで完了した作業を記述する。該当なしの場合はこのセクションを省略する。

- 変更したファイルとその内容
- 実行したコマンドとその結果

## 解決策の候補

検討中の解決策がある場合に記述する。該当なしの場合はこのセクションを省略する。

### 案1: <名前>

- 概要
- メリット / デメリット
- 参考: <URL やファイルパス>

### 案2: <名前>

...

## ユーザーの意向

調査中にユーザーが示した判断や好みがあれば記録する。（例: 「案2の方向で進めたい」「パフォーマンスより保守性を優先したい」）

該当なしの場合はこのセクションを省略する。

## 未解決の論点

まだ調査が必要な事項や、判断が必要な事項があれば記述する。

## 次のステップ

この文書を受け取ったエージェントが最初に行うべきアクションを明記する。（例: 「案1の方針でプランモードに入る」「〇〇について追加調査が必要」）

## 参考リンク

- [タイトル](URL)
```

### 注意事項

- ハンドオーバー文書は別セッションからの参照を前提とする。文脈を共有しない読者が理解できるよう、省略せずに記述する。
- 会話中にコードの変更を行っていた場合、その変更内容も文書に含める。
