# One-Pager

## なぜこの形式か

テキストや ASCII art で時系列・対応関係・前後関係・状態遷移・エンティティ関係を説明しようとすると、等幅フォントや端末幅に依存してすぐレイアウトが崩れ、要素間の関係が視覚的に追いにくくなる。HTML なら座標を明示的に指定できる SVG が使え、色・太さ・破線・矢印の向きといった手掛かりも豊富に使える。読み手は自分のブラウザで開き、ズームしたりリンクを辿ったりできる。

対象に次のような構造があるときは、ユーザーが「説明して」「まとめて」としか言っていなくてもこのスキルを積極的に提案する価値がある:

- 時系列 (何が起きた順に並ぶか)
- 対応関係・ペアリング (AのこれとBのこれが対応する、比較される)
- 前後関係・因果 (この状態からこの状態に変化した)
- 分岐・パターン比較 (複数の仮説/シナリオを並べて見せる)
- 状態遷移 (どの入力で状態が移るか、ガード条件は何か)
- エンティティ/モデル関係 (誰が誰を持ち、多重度はいくつか)
- パイプライン・データフロー (どのステージで何が起き、何が次に流れるか)

逆に対象が単純な箇条書きで済むもの (単なるリスト、単一の数値、はい/いいえ) なら、わざわざ HTML にする意味は薄い。素直にテキストで答える。

## 出力形式

**原則は single-file HTML**。CSS/JS はもちろん SVG も本文中にインラインで書き、外部ファイル参照なしで完結させる。

**外部ライブラリは使ってよい。** ブラウザで直接開くだけなので Anthropic の Artifact のような CSP 制約はない。Mermaid を CDN から読んで状態遷移図を書く、`<script src="https://cdn.jsdelivr.net/...">` で軽い可視化ライブラリを使う、といったことも自由。SVG を手書きするのが手間な形状 (複雑な有向グラフなど) では、こうしたライブラリが素直な選択になる。

**画像を含めたいときは single-file に囚われない。** 数十 KB 程度の小さい画像なら `<img src="data:image/png;base64,...">` として埋め込んでよいが、それより大きいものを無理に base64 化してファイルを肥大化させるのは本末転倒。画像で伝わるなら画像を使い、ファイル構成は `<name>/index.html` + `<name>/assets/` のようにディレクトリで括る。写真や撮影スクリーンショットが必要な題材では最初からディレクトリ構成にした方が素直。

**"ペライチ" は「読み手が1枚として読める」体験のこと**であって「1ファイル」の意味ではない。single-file にこだわった結果、画像を落として説明が伝わらなくなる方が失敗。

## 進め方

1. **説明したい内容の構造を先に決める。** 何が縦軸・横軸になるか、何と何を対応づけるか、どのノード/エッジ/ステージが必要かを整理してから座標や要素を組む。行き当たりばったりで rect を置き始めると、後から要素がぶつかったり文字が重なったりする。この整理段階で「対応関係」に見えたものが実は「状態遷移」だった、というようにお題の性質が見えてくることもある。**その結果として選ばれる構造 (トラック並置なのか、ノード-エッジ図なのか、ER 図なのか、レーン付きフローなのか) は、お題ごとに変わる。**
2. **[`assets/template.html`](assets/template.html) を土台にする。** ゼロから HTML を書く必要はない。テンプレートには CSS (見出し、callout、凡例、テーブル)、SVG の矢印マーカー定義、ダークテーマの配色トークンが用意済み。中身の `<h1>`/`<p>`/`<svg>` 要素を書き換えていく。テンプレの SVG サンプル (2 トラック対応関係) は「型の一例」なので、お題に合わなければ丸ごと差し替えてよい。
3. **保存先を決める。**
   - プロジェクトに scratchpad 的な置き場所の規約がある場合 (例: `.844196/` のような gitignore された個人ディレクトリ、`docs/` 配下の調査メモディレクトリなど) はそれに従う。
   - 特に規約がなければ、作業中のトピックに関連するディレクトリ、それも無ければカレントディレクトリに置く。
   - single-file なら `<説明が付く名前>.html`、ディレクトリ構成なら `<説明が付く名前>/index.html` + `<説明が付く名前>/assets/`。
   - ファイル新規作成の際にプロジェクト固有の手順 (例: 空ファイルを touch してから Read → Write する、等) があればそれに従う。
4. **書き終えたら `chroma` でブラウザに開く。**
   ```bash
   chroma "file://$(realpath path/to/one-pager.html)"
   ```
   ただし WSL2 で Windows 側のブラウザが既定になっている環境では、Linux 側のパスを含む `file://` URL は解決できない。`wslpath -m` で `//wsl.localhost/<distro>/...` 形式 (UNC を URL 互換に整えたもの) に変換して渡す:
   ```bash
   chroma "file:$(wslpath -m path/to/one-pager.html)"
   ```
   `wslpath` は WSL 上でしか存在しないコマンドなので、その有無で環境を分岐できる。`chroma` が使えない環境 (このユーザー固有のツールなので、無ければ普通に) では、ファイルパスを伝えてユーザーに自分で開いてもらう。
5. **チャットにも要点を書く。** 生成した HTML は補助教材であって代替物ではない。何を図示したか、結論は何かをチャット上でも簡潔に説明する。ページを開かせて終わりにしない。

## 図を組み立てるときのコツ

以下は SVG を手書きするときによく効くコツ。Mermaid など外部ライブラリを使う場合は該当しない項目もある (色の意味統一や凡例の必要性は共通)。

- **座標は手計算でよい。** 要素数が多くない一枚絵なら、`<rect>`/`<circle>`/`<line>`/`<text>`/`<polyline>`/`<path>` の座標を直接書き下ろす方式で十分な品質になり、この方が却って自由度が高く速い。D3.js のようなライブラリは、動的更新や大規模データが必要になってから検討する。
- **`viewBox` を使う。** `<svg width="900" height="240" viewBox="0 0 900 240">` のようにしておくと、後で全体のサイズ調整がしやすい。
- **色は「意味」に固定して一貫させる。** 例えば「片方の系列は青系、もう片方は赤系」「確定した事実は緑、未検証の仮説は赤/黄」「入力状態と出力状態で色を分ける」のように、ページ全体で同じ色が同じ意味を持つようにする。都度その場の思いつきで色を変えない。テンプレートの callout (`.info`/`.warn`/`.ok`) と合わせて統一すると分かりやすい。
- **関係は矢印/線で明示する。** 対応・遷移・依存を示したいときは、位置を揃えて並べるだけでなく `<line marker-end="url(#arrow)">` や `<path>` で明示的に結ぶと格段に伝わりやすい。テンプレートに緑/赤の矢印マーカー定義がある。
- **ラベルは要素の中か直下に置く。** `text-anchor="middle"` で図形の中心にラベルを置くのが基本。長い注釈は図の外の `<figcaption>` に逃がして SVG 内は短いラベルだけにする方が読みやすい。
- **凡例を忘れない。** 色や線種に意味を持たせたら、テンプレートの `.legend` ブロックで凡例を必ず添える。作った本人には自明でも、見る側には自明ではない。
- **要素がぶつかったら座標を計算し直す。** テキストが重なったり矢印が図形を突き抜けたりしていないか、書いた後に座標を目で追って確認する。
- **SVG は後に描いた要素が前の要素を覆い隠す。** 特に「見出しラベルの右に長い注釈テキストを続けて置き、その下 (同じ縦範囲) に矩形を描く」パターンでは、矩形がラベルの続きを隠して文字が途中で切れる事故が起きやすい。ラベル行と図形の行は縦方向に十分離す (ラベルの `y` と、その下の図形の `y` の間に最低でも文字の高さ+余白分の差を空ける) か、ラベルの右端が図形の `x` 開始位置を超えないよう文字数を見積もっておく。

## 参考実装は「型」として見る、真似するのではなく

[`references/example-frame-pairing.html`](references/example-frame-pairing.html) に、2つの時系列データの対応関係のズレを図示した実例を収録している。テーブルと SVG の併用、複数シナリオの並置、色分けのルールなど、具体的な組み立て方の一例。

**ただし、これはあくまで「2 トラック対応関係」というお題への一解でしかない。** 状態遷移図、ER 図、パイプライン図、階層ツリー、マトリクス比較、決定木 —— お題の構造が違えば適切なレイアウトも違う。参考実装の見た目 (上下 2 トラック + 縦の対応矢印) に無理やり寄せると、本来なら丸で状態を書いてラベル付き矢印で遷移を描くべきものが、意味の合わない矩形の並びになってしまう。

参考実装から借りるべきものは以下:

- テンプレートの CSS/配色/callout の使い方
- 色と意味を最初に宣言する (`div.callout.info` で凡例を書いておく) というスタイル
- 凡例 (`.legend`) と `<figcaption>` の使い分け
- SVG に短いラベル、外側に長い説明という役割分担

一方で以下は毎回お題に合わせて選び直す:

- 全体レイアウト (2 トラック並置なのか、円環状の遷移なのか、格子状のマトリクスなのか)
- 図形の意味付け (矩形は何を表すのか、色は何の軸なのか)
- どこにテーブルを置くか、そもそもテーブルが要るか

参考実装のパターンにお題を押し込めるのではなく、お題から出発してレイアウトを設計し、必要な CSS/矢印マーカー等の部品だけをテンプレートと参考実装から借りてくる、という順序で組む。
