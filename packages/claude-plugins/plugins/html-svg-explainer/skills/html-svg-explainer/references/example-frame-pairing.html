参考実装: 人力メモ vs 自動文字起こし(ASR)の発言対応がズレる問題

これは html-svg-explainer スキルの組み立て方を示すための架空の題材。 会議で人間が取った発言メモ (どの発言が何番目か) と、自動文字起こし (ASR) が出力した発言単位が 途中から1つずつズレる、という調査の図解を想定している。 題材そのものよりも「時系列 + 対応関係 + 複数シナリオ比較」をどう SVG に落とすかの型を見てほしい。

色の意味: 緑 = 実測で確定した事実、 赤 = まだ仮説 (検証未完)。以降このページ全体で共通。

1. 実測で確定した事実

1-1. 発言数の食い違い

項目人力メモASR出力
総発言数142139
1発言に2人分の音声が混在した回数-3
発言が2つに分割された回数-0

ASR は3箇所で「2人の発言を1つにまとめて」出力しており、 その結果 142 発言 → 139 発言になっている。分割 (1つの発言を2つに割ってしまう) は確認されなかった。

1-2. 発言 #38 付近の具体例

#人力メモASR出力
37杏山: 予算の件ですが杏山: 予算の件ですが
38宇沢: それ賛成です!宇沢: それ賛成です!あと日程も見直したいです! ← 2発言が結合
39宇沢: あと日程も見直したいです!
40杏山: 了解です杏山: 了解です

人力メモでは「賛成です」と「日程も見直したいです」を 別の発言 (2ターン) として記録しているが、 ASR は無音区間が短かったためか 1発言として結合 して出力している。 これ以降、ASR側の発言番号は人力メモより1つ小さいまま推移する。

2. なぜ結合が起きるのか (3つの仮説シナリオ)

確定していること: ASR が人力メモより発言数が少ないのは 複数発言の結合が原因であり、 語の誤認識による欠落ではない (文字起こしの内容自体は結合後も両方の発言が含まれている)。
ただし: 結合が起きる条件 (無音区間の長さで判定しているのか、話者交代を検知できていないのか) は 未確定。ログの無音区間長を実測してさらに切り分けが必要。

3. まとめ

今回の学び: 2つの時系列データ (人力メモ/ASR) を比較するときは、単に件数を突き合わせるだけでなく、 「どの時点から」「どの要素とどの要素が」対応しなくなったかを1件レベルで特定して図示すると、 原因の仮説がぐっと立てやすくなる。
